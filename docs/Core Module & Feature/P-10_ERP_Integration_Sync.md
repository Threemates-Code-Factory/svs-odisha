# P-10 · School ERP Integration and Sync

**Module Code:** P-10  
**Priority:** 🟢 Medium  
**Estimated Effort:** 6–8 Weeks  
**Team:** 1 Senior Backend Developer (API/Integration) + 1 QA  
**Depends On:** P-01 (Data Validation), P-11 (RBAC), Core DB Schema  

---

## 1. Module Overview

Individual SSVM schools may operate standalone school management systems (local ERPs) entirely separate from the SVS Head Office EMS. The current ASP.NET Web Forms system has no API layer whatsoever, forcing school staff to manually enter the same data into both their local system and the SVS EMS. This causes data duplication, inconsistency, and unreliable Head Office reporting.

This module builds a **standardised ERP Integration Layer** — an open API platform that any school's local system can connect to for bi-directional data sync.

---

## 2. Functional Requirements

### 2.1 Open API Specification

Publish a REST API (OpenAPI 3.0 spec) that any SSVM local ERP can use to:
- Push data to SVS EMS (student admissions, fee collections, staff updates)
- Pull data from SVS EMS (syllabus publications, exam schedules, training calendar)
- Receive event notifications (webhooks)

**API Authentication:**
- Each SSVM receives a unique API Key (generated by Super Admin)
- API Key passed in Authorization header
- Rate limiting: 1,000 requests/hour per SSVM key
- All API traffic logged to the sync audit log

### 2.2 Supported Sync Operations

**Push (Local ERP → SVS EMS):**

| Resource | Operation | Validation Applied |
|----------|-----------|--------------------|
| Student Admission | Create / Update / Delete | P-01 rules (Aadhaar uniqueness, required fields) |
| Student Status Change | Update | Valid status values only |
| Staff Record | Create / Update | P-01 rules (designation from master, Aadhaar uniqueness) |
| Fee Collection | Create | Amount validation, student must exist |
| Exam Attendance | Create/Update | Student must be registered for the exam |

**Pull (SVS EMS → Local ERP):**

| Resource | Description |
|----------|-------------|
| Syllabus Publications | Latest syllabus documents for the school's class range |
| Exam Schedule | Upcoming exam events |
| Training Calendar | Training programmes open for school's staff |
| Circular / Announcements | Head Office circulars |

**Webhooks (SVS EMS pushes events to Local ERP):**

| Event | Payload |
|-------|---------|
| `publication.created` | New syllabus/circular published |
| `exam.scheduled` | New exam event created for the school's classes |
| `training.slot.available` | Training programme with open enrollment |
| `compliance.violation` | Compliance violation detected for this SSVM |

### 2.3 Data Standardisation

The API enforces a canonical data format:

- **Student Name:** Title case, no special characters except hyphens and apostrophes
- **Class values:** Must match master list (Praramva, Bodh, I–XII) — mapped from local system's values using a configurable mapping table
- **Aadhaar:** 12 digits, numeric only
- **Mobile:** 10 digits, numeric only, no country code
- **Designation:** Must match SVS master designation list — mapping table provided
- **Date format:** ISO 8601 (YYYY-MM-DD) only

If a push request fails data standardisation, the API returns a 422 error with field-level error details.

### 2.4 Data Sync Schedule

**Modes:**
- Real-time (event-driven): Local ERP calls the API immediately on data change
- Scheduled batch: Local ERP sends a batch of changes at a defined interval (hourly, daily)
- Manual trigger: SSVM admin can trigger a full sync from the Data Quality Dashboard

**Sync Log:**
Every sync operation is logged:
- Sync ID, SSVM code, operation type, resource, record ID
- Request payload (sanitised — no Aadhaar in plain text in logs)
- Response status code and message
- Timestamp

### 2.5 Data Quality Dashboard (SSVM)

An SSVM admin can view the sync health of their school:

- Last sync time
- Records synced this week (by resource type)
- Failed syncs with error details and retry option
- Data quality score: % of records in SVS EMS that match their local system
- Discrepancy list: records that exist in one system but not the other

### 2.6 Developer Integration Kit

Published publicly for local ERP vendors:
- OpenAPI 3.0 YAML specification
- Postman collection with example requests
- SDK snippets for common languages (Python, JavaScript, PHP)
- Sandbox environment for testing without affecting production data
- Webhook testing tool (Webhook.site-style endpoint for local ERP developers)

## 3. Data Models

### 3.1 API Keys Table
```
api_keys
├── key_id (UUID, PK)
├── ssvm_code
├── api_key_hash (SHA-256 of key — never store plaintext)
├── created_by
├── created_at
├── last_used_at
├── is_active (BOOL)
└── rate_limit_per_hour
```

### 3.2 Sync Log Table
```
sync_log
├── log_id (UUID, PK)
├── ssvm_code
├── api_key_id (FK)
├── operation (ENUM: PUSH, PULL, WEBHOOK)
├── resource
├── record_id (nullable)
├── http_method
├── status_code
├── error_message (nullable)
├── request_size_bytes
├── response_time_ms
└── logged_at
```

### 3.3 Class Mapping Table
```
ssvm_class_mapping
├── mapping_id (UUID, PK)
├── ssvm_code
├── local_class_value
├── svs_class_value
└── created_at
```

## 5. Acceptance Criteria

- [ ] A student admission POSTed via API is validated (P-01 rules) and reflected in the SVS EMS within 10 seconds
- [ ] An API request with an invalid API key returns 401
- [ ] An API request exceeding 1,000/hour rate limit returns 429
- [ ] A webhook for `publication.created` is delivered to the registered URL within 60 seconds of the publication being created
- [ ] Data Quality Dashboard shows sync discrepancies between local and SVS EMS data
- [ ] Class mapping table correctly translates a local class name "Grade 5" to SVS canonical "V"

---

## 6. Developer Notes

- Never log API keys in plaintext — hash before storing and compare hashes
- Webhook delivery must have a retry policy: attempt delivery 3 times with exponential backoff (1 min, 5 min, 15 min) before marking as failed
- Sandbox environment should use a separate database with anonymised sample data — never share production data in sandbox
- The API should be versioned from day one (/api/v1/...) — this allows breaking changes in future without disrupting connected ERPs
